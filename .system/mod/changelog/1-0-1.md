# KrepAPI 1.0.1

## Problems Fixed

- **ByteBuffer with Fixed-Size Allocation — No Bounds Check**: `ByteBuffer.allocate(65536)` is used for bindings and `512` for Hello/ClientInfo. If a payload grows larger, there's a silent `BufferOverflowException` crash. No dynamic growth, no try-catch.
- **c2s_key_action Handler is Empty**: The receiver for `KrepapiKeyActionC2SPayload` is completely empty — `registerGlobalReceiver(..., (payload, context) -> {})`. Key presses on the Fabric server are silently ignored.
- **No Input Validation in `ProtocolBuf.readUtf()`**: Without a length limit on UTF string reads, a malicious client can send arbitrarily long strings in `actionId` or `modVersion`, leading to uncontrolled heap growth.
- **Dispatch Doesn't Cancel on First Match**: In `KrepapiKeyPipeline.dispatch()`, `consume = true` is set when a listener returns `true`, but all subsequent listeners still run. The variable is never reset.
- **volatile Fields Instead of AtomicReference / Final Config**: `requireClientOnDedicatedServer`, `minimumModVersion`, and `handshakeTimeoutTicks` are all `volatile`. Stylistically questionable compared to a clean config class.
- **Lexicographic Version Comparison**: With `minimum-mod-version: "1.9"` and then `"1.10"` — lexicographically `"1.9" > "1.10"`. No real SemVer parser.
- **HANDSHAKE_TICKS Map Doesn't Clean Up Reliably**: Entry only removed on `e.answered` or timeout. Rapid joins from unmodded clients could grow the map.
- **No Unit Tests**: The `:protocol` module would be ideal for encode/decode roundtrip tests.
- **Release Phase Not Wired Up**: `KeyAction.PHASE_RELEASE` is defined and transmitted but the client-side `KeyMapping` never fires a release event.
